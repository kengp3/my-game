## Summary

This phase introduced real game data into the wiki, populating the VitePress site with story chapter pages (prologue through chapter 12), beginner combat guides, one boss page, one equipment page, and one map area page. The `StoryChapterTemplate.vue` component and the VitePress config (`config.ts`) were also reviewed.

Overall the content is well-structured and the VitePress configuration is clean. The most significant technical issues are:

1. A `ChoicePoint.vue` component is imported in `StoryChapterTemplate.vue` but no matching file was included in the reviewed set — its existence cannot be confirmed.
2. Several nav links in `config.ts` point to sidebar sections (`/guide/characters/`, `/guide/systems/`) that have no corresponding content files, which will result in 404 pages.
3. Five chapter files (chapters 5, 6, 8, 10, 11) use `areas: []` (empty array), which causes the component to render a placeholder message — acceptable for a stub, but warrants attention as the wiki matures.
4. Minor frontmatter inconsistency: `chapter-prologue.md` uses `chapterNumber: 0` while the index page lists it separately as "序幕" (prologue), not a numbered chapter — this is benign but could confuse consumers of the frontmatter field.

No security issues were found. The codebase contains no hardcoded secrets, no injection risks, and no unsafe patterns.

## Warnings

### WR-01: Unverified ChoicePoint Component Import

**File:** `.vitepress/theme/components/StoryChapterTemplate.vue:65`
**Issue:** `ChoicePoint.vue` is imported and used in `StoryChapterTemplate.vue`, but this file was not included in the reviewed file set. If `ChoicePoint.vue` does not exist or its `title` / `choices` prop names differ from what is passed here, every story chapter page will fail to render at build time.

The call site passes:
```vue
<ChoicePoint
  v-for="(choice, index) in frontmatter.choices"
  :key="index"
  :title="choice.title"
  :choices="choice.options"
/>
```

If `ChoicePoint` expects a different prop name (e.g., `options` instead of `choices`), all chapters that populate `choices` in frontmatter will silently render nothing or throw a Vue warning.

**Fix:** Verify that `.vitepress/theme/components/ChoicePoint.vue` exists and that its props are named `title` and `choices`. If the prop is named `options`, update this call to `:choices="choice.options"` → `:options="choice.options"`.

### WR-02: Nav Links to Sections With No Content Files

**File:** `.vitepress/config.ts:26,29`
**Issue:** Two top-level nav items link to sections that have no content pages in this review set — and whose sidebars contain only a single self-referencing link with no real page behind it:

- `{ text: '角色職業', link: '/guide/characters/' }` — sidebar entry points only to `/guide/characters/` itself, but no `guide/characters/index.md` was reviewed.
- `{ text: '系統機制', link: '/guide/systems/' }` — same pattern.

Visiting either URL in production will produce a 404 unless the index files exist on disk.

**Fix:** Either create stub `index.md` files in `guide/characters/` and `guide/systems/`, or remove those nav entries until the sections have content. Example stub:

```markdown
---
layout: doc
title: 角色職業
---

# 角色職業

::: info 資料待補充
本章節資料尚未收集完整，待後續補充。
:::
```

### WR-03: `example-boss.md` Uses Non-Standard Frontmatter Key `name` Instead of `title`

**File:** `guide/bosses/example-boss.md:3`
**Issue:** The frontmatter uses `name: 裂角凱洛克（Kailok the Hornsplitter）` (not `title:`). VitePress uses `title` for `<title>` tag generation and breadcrumb display. Because `layout: doc` is set and `title` is absent, VitePress will fall back to deriving a page title from the first `<h1>` — but `BossTemplate` (a component) produces the `<h1>`, not the Markdown body. The resulting browser tab title will be the site name with no page-specific title, and search indexing will be degraded.

The same pattern appears in `guide/equipment/example-equipment.md:3` and `guide/maps/example-area.md:3`.

**Fix:** Add a `title:` key to each of these frontmatter blocks:

```yaml
---
layout: doc
title: 裂角凱洛克（Kailok the Hornsplitter）
name: 裂角凱洛克（Kailok the Hornsplitter）
...
---
```

## Info

### IN-01: `chapterNumber: 0` on Prologue May Cause Display Quirk

**File:** `guide/story/chapter-prologue.md:4`
**Issue:** `StoryChapterTemplate.vue` renders the badge as `第 {{ frontmatter.chapterNumber }} 章` when `chapterNumber` is truthy. Since `0` is falsy in JavaScript, the badge is suppressed for the prologue — this is actually the correct visual result, but the intent of the `v-if="frontmatter.chapterNumber"` condition being used to skip the prologue badge is non-obvious. A future author setting `chapterNumber: 0` on a different page expecting a "第 0 章" badge would be surprised. This is currently benign.

**Fix:** Consider using `v-if="frontmatter.chapterNumber != null"` in the component if explicit zero-numbered chapters are ever needed, or document the convention that prologue pages should omit `chapterNumber` entirely.

### IN-02: `chapter-prologue.md` Uses `prev: false` But No Other Chapters Set `next: false`

**File:** `guide/story/chapter-prologue.md:16`
**Issue:** The prologue sets `prev: false` to suppress the "previous page" nav link, but no chapter sets `next: false` on the final page (chapter-12). This is inconsistent and may result in VitePress auto-generating a broken "next" link on the last chapter pointing to whatever page follows in the sidebar order.

**Fix:** Add `next: false` to `guide/story/chapter-12.md` frontmatter for consistency:
```yaml
next: false
```

### IN-03: `chapter-2.md` Missing `prev: false` / Navigation Consistency

**File:** `guide/story/chapter-2.md` (frontmatter)
**Issue:** `chapter-prologue.md` sets `prev: false`, and `chapter-1.md` sets `prev: false`. However `chapter-2.md` does not set `prev`. This is inconsistent — chapters 1 and prologue suppress "prev" but chapter 2 does not, which means chapter 2 will auto-link backwards to chapter 1 (which may be intentional) while the prologue has no backwards link. There is no structural bug here, but the pattern is inconsistent and may cause confusing prev/next nav as more chapters are added.

**Fix:** Decide on a consistent convention: either use VitePress sidebar order for all prev/next navigation (remove all `prev: false` overrides), or set explicit `prev`/`next` links on all chapter pages.

### IN-04: Magic Number in Search Boost Config

**File:** `.vitepress/config.ts:143`
**Issue:** The MiniSearch boost values `{ title: 4, text: 2, titles: 1 }` are inline magic numbers with no comments explaining the rationale. This is a minor maintainability concern — someone tuning search later has no context for why these values were chosen.

**Fix:** Add a comment:
```ts
// Boost title matches heavily (4x), body text moderately (2x), breadcrumb titles lightly (1x)
boost: { title: 4, text: 2, titles: 1 }
```
