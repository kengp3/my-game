1. Create `.github/workflows/deploy.yml` with the following exact content. IMPORTANT: The branch trigger MUST be `master` (not `main`) because `git branch` shows the current branch is `master`. Note: D-08 in CONTEXT.md mentions "推送 main 分支即部署" as a conceptual description; the actual git environment uses `master`, so the workflow targets `master` accordingly.

```yaml
name: Deploy VitePress to GitHub Pages

on:
  push:
    branches: [master]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Setup Pages
        uses: actions/configure-pages@v4

      - name: Install dependencies
        run: npm ci

      - name: Build with VitePress
        run: npm run docs:build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: .vitepress/dist

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Key details per RESEARCH.md:
- Branch is `master` (verified from `git branch` output), NOT `main` — CONTEXT.md D-08 references "main" conceptually, but the actual local git branch is `master`; workflow uses `master` to match the real environment
- Uses `actions/checkout@v4`, `actions/setup-node@v4`, `actions/configure-pages@v4`, `actions/upload-pages-artifact@v3`, `actions/deploy-pages@v4` -- all latest stable versions
- Node version 20 (LTS, stable; RESEARCH.md notes Node 20 is preferred over 24)
- `npm ci` (not `npm install`) for reproducible builds from lockfile (per threat T-01-03)
- Artifact path `.vitepress/dist` (content root is repo root, not `/docs`)
- `permissions` block explicitly limits to `contents: read`, `pages: write`, `id-token: write` (principle of least privilege)
- `concurrency` group prevents overlapping deployments

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner -> GitHub Pages | Build artifacts uploaded from CI runner to Pages hosting |
| npm registry -> CI runner | Dependencies installed from npm during CI build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | GitHub Actions workflow permissions | mitigate | Workflow uses principle of least privilege: `contents: read` (no write to repo), `pages: write` (only Pages), `id-token: write` (required for Pages deployment token). No `GITHUB_TOKEN` write access. |
| T-03-02 | Tampering | npm supply chain in CI | mitigate | Use `npm ci` (installs from lockfile only, fails if lockfile out of sync). Pin Node version to LTS 20. `actions/setup-node@v4` with `cache: npm` uses official action. |
| T-03-03 | Elevation of Privilege | workflow_dispatch trigger | accept | `workflow_dispatch` allows manual triggering by repo collaborators. This is intentional for manual redeploys. Repo access controls gate who can trigger. |
| T-03-04 | Denial of Service | Concurrent deployments | mitigate | `concurrency: { group: pages, cancel-in-progress: false }` prevents overlapping deployments that could cause partial uploads. |
| T-03-05 | Information Disclosure | Build output contents | accept | VitePress dist output is the public website. No secrets in build output. `.planning/` directory is not included in VitePress build output (only Markdown content root is built). |
</threat_model>
