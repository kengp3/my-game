## 關鍵 Bug：`withBase()` 缺失

### 問題說明

[VERIFIED: VitePress 官方文件 + 直接程式碼審查 2026-04-11]

**VitePress 文件明確說明：** Markdown 內的靜態圖片路徑會自動套用 base，但 Vue 元件中的動態綁定（`:src="..."` 由 frontmatter 或 theme config 提供）**必須手動呼叫 `withBase()`**。

本站設定：
```typescript
// .vitepress/config.ts
base: '/my-game/',
```

三個模板目前的寫法（有問題）— 已於 2026-04-11 直接驗證：
```vue
<!-- CharacterTemplate.vue L13, BossTemplate.vue L17, EquipmentTemplate.vue L17 -->
<img :src="frontmatter.image" />
```

正確寫法：
```vue
<img :src="withBase(frontmatter.image)" />
```

**不修正的後果：** 本地 `npm run docs:dev` 時圖片可能正常顯示（dev server 不加 base），但部署至 GitHub Pages（`/my-game/`）後，圖片請求會打到 `https://user.github.io/images/kliff.webp`（404）而非正確的 `https://user.github.io/my-game/images/kliff.webp`。

### 修正位置

| 檔案 | 行號 | 現行 | 修正後 |
|------|------|------|--------|
| `.vitepress/theme/components/CharacterTemplate.vue` | L13 | `:src="frontmatter.image"` | `:src="withBase(frontmatter.image)"` |
| `.vitepress/theme/components/BossTemplate.vue` | L17 | `:src="frontmatter.image"` | `:src="withBase(frontmatter.image)"` |
| `.vitepress/theme/components/EquipmentTemplate.vue` | L17 | `:src="frontmatter.image"` | `:src="withBase(frontmatter.image)"` |

`withBase` 已在所有三個檔案中 import（`import { useData, withBase } from 'vitepress'` — 確認於程式碼審查），因此只需更新模板屬性，無需新增 import。

## Common Pitfalls

### Pitfall 1: `withBase()` 未套用導致部署後圖片 404
**What goes wrong:** 本地開發正常，但 GitHub Pages 部署後圖片全部 404
**Why it happens:** VitePress dev server 不套用 base，但 production build 會；`:src="frontmatter.image"` 不會自動加上 `/my-game/` 前綴
**How to avoid:** 在三個模板中改為 `:src="withBase(frontmatter.image)"`，並在 `npm run docs:build` 後以 `npm run docs:preview` 驗證
**Warning signs:** 本地 dev 圖片正常但 preview 顯示 404

### Pitfall 2: Fextralife thumbnail URL 縮略圖取錯
**What goes wrong:** 使用 `/thumb/` URL 下載到小圖（85px），轉換後太糊
**Why it happens:** Fextralife wiki 對 thumbnail 用特殊 URL 格式（`/thumb/a/ab/name.png/85px-name.png`）
**How to avoid:** 去除 URL 的 `/thumb/` 段和結尾的 `/85px-...` 部分取得原始圖片
  - Thumbnail: `.../thumb/3/30/Sword_of_the_wolf_weapons_crimson_desert_wiki_guide_150px.png/85px-...`
  - Original: `.../3/30/Sword_of_the_wolf_weapons_crimson_desert_wiki_guide_150px.png`
**Warning signs:** 下載的圖片寬度小於目標尺寸

### Pitfall 3: Fextralife CDN 拒絕 curl 請求
**What goes wrong:** `curl` 下載圖片失敗（403 或空檔案）
**Why it happens:** CDN 可能有 User-Agent 或 Referer 限制
**How to avoid:** 加入 `-H "Referer: https://crimsondesertgame.wiki.fextralife.com/"` 和 `-A "Mozilla/5.0"` 標頭
**Warning signs:** curl 回傳 403 或空檔案

```bash
curl -L -o output.png \
  -H "Referer: https://crimsondesertgame.wiki.fextralife.com/" \
  -A "Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7)" \
  "https://static0.fextralifeimages.com/file/crimsondesertgame/..."
```

### Pitfall 4: `public/` 目錄不存在
**What goes wrong:** `sharp` 寫入時報錯或圖片消失
**Why it happens:** `public/images/characters/` 等目錄尚未存在
**How to avoid:** 在寫入圖片前先 `mkdir -p public/images/{characters,bosses,equipment}`

### Pitfall 5: `sharp` 在 ESM 模組環境的 import 問題
**What goes wrong:** `import sharp from 'sharp'` 在 `type: "module"` 的 package.json 中報錯
**Why it happens:** Sharp 0.34.x 為 CJS，在純 ESM 環境需留意
**How to avoid:** 若用 script 方式執行，使用 `.cjs` 副檔名或 `require()`；或以 `import()` 動態載入；或直接在 package.json scripts 中以 `node --input-type=commonjs` 執行
**Alternative:** 直接使用 `brew install imagemagick && magick` 避開此問題

```javascript
// Source: sharp.pixelplumbing.com/api-output
const sharp = require('sharp')
const path = require('path')

const CHAR_WIDTH = 600
const EQUIP_WIDTH = 400
const QUALITY = 80

async function convert(src, dest, maxWidth) {
  await sharp(src)
    .resize({ width: maxWidth, withoutEnlargement: true })
    .webp({ quality: QUALITY })
    .toFile(dest)
  console.log(`Done: ${path.basename(dest)}`)
}

async function main() {
  // 角色（600px）
  await convert('/tmp/kliff.png', 'public/images/characters/kliff.webp', CHAR_WIDTH)
  await convert('/tmp/damiane.png', 'public/images/characters/damiane.webp', CHAR_WIDTH)
  await convert('/tmp/oongka.png', 'public/images/characters/oongka.webp', CHAR_WIDTH)

  // Boss（600px）
```

### Dimension 2 — Format and Constraints

**What to check:** Each WebP file has correct format, is within max-width bounds, and has a sane file size (not 0 bytes, not suspiciously large for a wiki card image).

**How to check (requires sharp or file utility):**
```bash
# Verify WebP magic bytes for each file (first 4 bytes: RIFF + size + WEBP)
for f in public/images/**/*.webp; do
  head -c 12 "$f" | xxd | grep -q "5249 4646" && echo "OK: $f" || echo "NOT_WEBP: $f"
done

# Check file sizes are non-zero and within sanity bounds (< 500KB each)
for f in public/images/**/*.webp; do
  size=$(wc -c < "$f")
  if [ "$size" -eq 0 ]; then echo "EMPTY: $f"
  elif [ "$size" -gt 512000 ]; then echo "TOO_LARGE (${size}B): $f"
  else echo "OK (${size}B): $f"
  fi
done
```

**Width verification via sharp (node one-liner):**
```bash
# characters and bosses must be <= 600px wide
node -e "
const sharp = require('sharp');
```
