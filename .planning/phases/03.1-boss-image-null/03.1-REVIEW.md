## Warnings

### WR-01: convert-images.cjs — partial output file left on sharp failure

**File:** `scripts/convert-images.cjs:49`
**Issue:** When `sharp().toFile(dest)` rejects mid-write (e.g., disk full, corrupt source), the output `.webp` file may already have been created but be incomplete. The outer `catch` logs the error and increments `failed`, but does not delete the partial `dest` file. A subsequent run of `verify-images.mjs --files` will pass the existence check and the RIFF magic-bytes check (WebP container header is written first) on a file that is corrupt or zero-padded beyond the header.

**Fix:**
```js
async function convertOne(src, dest, maxWidth) {
  if (!fs.existsSync(src)) {
    throw new Error(`Source missing: ${src}`);
  }
  fs.mkdirSync(path.dirname(dest), { recursive: true });
  try {
    await sharp(src)
      .resize({ width: maxWidth, withoutEnlargement: true })
      .webp({ quality: QUALITY })
      .toFile(dest);
  } catch (err) {
    // Remove partial output so subsequent verify runs don't false-pass
    try { fs.unlinkSync(dest); } catch (_) {}
    throw err;
  }
  const size = fs.statSync(dest).size;
  console.log(`OK: ${path.basename(dest)} (${size} bytes, max ${maxWidth}px)`);
}
```

### WR-02: verify-images.mjs — dist HTML bare-URL negative check can produce false failures

**File:** `scripts/verify-images.mjs:129-131`
**Issue:** The negative assertion at line 129 constructs `bareUrl = '/images/characters/kliff.webp'` and builds a regex to check that no `src=` attribute with that exact bare path appears in the rendered HTML. However, VitePress bundles all page HTML into a single `.html` file that may include inlined component code (or serialised SSR state) from *other* pages that have not yet had their images updated. A bare `/images/…` reference from a different page's partial could appear in the same HTML blob, triggering a spurious FAIL unrelated to the page under test.

**Fix:** Scope the negative check to match only within the `<img>` tag that corresponds to the frontmatter image field, or narrow the regex to require the slug to be unique per file by checking `src="${bareUrl}"` only within the `<div class="*-image"` block. At minimum, add a comment warning that the check is page-level, not component-level, so future maintainers understand its limits:

```js
// NOTE: This is a page-level check. If another template on the same page
// legitimately embeds a different bare URL, this assertion may false-fail.
// Acceptable for Phase 03.1 where each page has exactly one image.
```

### WR-03: verify-images.mjs — `--dist` flag not wired into runAll path guard

**File:** `scripts/verify-images.mjs:138-142`
**Issue:** `checkDist()` requires a completed `docs:build` to have run first. When `node scripts/verify-images.mjs` is invoked without flags (`runAll = true`), `checkDist()` is included and will immediately FAIL if no build has been run yet — even on a clean developer machine. The VALIDATION.md spec says `--dist` should only run as part of the full suite (`npm run docs:build && node scripts/verify-images.mjs`), but `runAll` silently includes it.

**Fix:** Exclude `--dist` from `runAll` (it requires an explicit opt-in) to match the documented usage contract:

```js
if (runAll || flags.has('--files'))       checkFiles();
if (runAll || flags.has('--widths'))      await checkWidths();
if (runAll || flags.has('--frontmatter')) checkFrontmatter();
if (runAll || flags.has('--templates'))   checkTemplates();
if (flags.has('--dist'))                  checkDist(); // never in runAll — requires prior build
```

### IN-01: verify-images.mjs — size threshold (500KB) inconsistent with VALIDATION.md spec (200KB)

**File:** `scripts/verify-images.mjs:27`
**Issue:** `MAX_FILE_BYTES = 500 * 1024` (500KB). The VALIDATION.md Wave 0 requirements specify `< 200KB` as the size threshold for each WebP file. At quality=80 and max 600px width, WebP output should be well under 200KB; the looser 500KB guard will not catch unexpectedly large outputs.

**Fix:** Align with the spec:
```js
const MAX_FILE_BYTES = 200 * 1024; // Dimension 2: < 200KB per VALIDATION.md Wave 0
```

### IN-02: verify-images.mjs — `--widths` check is async but not excluded from runAll sequencing concern

**File:** `scripts/verify-images.mjs:139`
**Issue:** `checkWidths()` is the only async check and is correctly `await`ed. However, if `sharp` is not installed (e.g., CI environment without devDependencies), the dynamic `import('sharp')` at line 55 will throw an unhandled rejection inside the IIFE. The outer `failures` counter will not be incremented; the process may exit 0 with a misleading `VERIFY PASSED` message if `--widths` is the only failing check.

**Fix:** Wrap the `checkWidths()` call site in a try/catch that routes to `fail()`:
```js
try {
  await checkWidths();
} catch (err) {
  fail(`checkWidths() threw: ${err.message}`);
}
```
